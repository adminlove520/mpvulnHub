#  Synology Active Backup for Microsoft 365 高危漏洞剖析  
 独眼情报   2025-06-30 05:31  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/KgxDGkACWnQUK0MhiaaMRrOV03cybCWw834MDXHepeGVQkVs6JiafMgaHcZTXOXmyu0wEL7mGlN7xMN0ZmSGaaWw/640?wx_fmt=png&from=appmsg "")  
  
  
6 月 27 日，瑞士安全公司 Modzero 披露了对 Synology Active Backup for Microsoft 365（简称 ABM）的一项严重漏洞研究：**一条被硬编码并在安装向导中泄露的 OAuth 客户端密钥，足以让任何人横跨租户、读取受害组织全部 Microsoft Teams 频道消息与 Microsoft 365 组数据**  
。

在管理员为 NAS 配置备份时，浏览器被重定向至 SynoOauth；返回 NAS 的 302 响应里含有如下参数（节选并脱敏）：  

```
&client_id=b4f234da-3a1a-4f4d-a058-23ed08928904
&client_secret=ARI8Q~sHOuwMoX.[...]

```

  
攻击者只需抓包一次，即可得到 **ABM 全局应用的 client_secret**  
。随后使用 OAuth 客户端凭据流：  

```
curl -X POST https://login.microsoftonline.com/<受害租户ID>/oauth2/v2.0/token \
  -d &#34;grant_type=client_credentials&#34; \
  -d &#34;client_id=b4f234da-3a1a-4f4d-a058-23ed08928904&#34; \
  -d &#34;client_secret=<泄露密钥>&#34; \
  -d &#34;scope=https://graph.microsoft.com/.default&#34;

```

  
返回的 JWT 里直接携带 
```
Group.Read.All
```

  
、
```
ChannelMessage.Read.All
```

  
 等权限，可无交互读取租户全部 Teams 频道与组信息。  
## 二、影响面与利用思路  
1. **安装量巨大**  
Synology 官方下载量超过 120 万；每一次安装过程都曾暴露相同密钥。  
  
1. **零前置权限**  
攻击者不需要是租户成员，只要掌握密钥，就能针对任意使用 ABM 的租户调用 Microsoft Graph API。  
  
1. **数据类型**  
  
1. Teams 频道（包括私有频道）全部消息内容  
  
1. 所有 Microsoft 365 组及其 Outlook 讨论、日历、附件等  
  
1. **潜在威胁场景**  
  
1. 社会工程、内网渗透的情报收集  
  
1. 供应链式勒索前置侦察  
  
1. 定向情报/竞业监视

## 五、教训与启示  
- **多租户 SaaS 的最小权限原则**  
应用在租户层面申请 
```
*.Read.All
```

  
 级别权限前，应接受独立安全评估；必要时拆分为细粒度权限集。  
  
- **供应链责任共担**  
云供应链漏洞往往由厂商“静默修复”，受害方却无法得知是否被利用；安全团队需主动获取 SBOM / 安全通告，并开启日志长期留存。  
  
- **中间件薄弱环节**  
OAuth 转发服务泄露敏感参数是常见失误，应在响应头、URL、日志等处消除密钥与令牌。

## 总结  
  
**一句话总结：**  
一旦把备份钥匙交给第三方，就要准备好某天钥匙可能流落黑市。Synology ABM 案例再次提醒我们，**云端“免费”备份服务若失守，其造成的横向、跨租户隐患会远超本地单点失陷**  
。  
  
对于安全从业者，最佳做法不是简单弃用，而是在选择任何备份/同步 SaaS 之前，将其权限、日志与应急预案清晰纳入威胁模型，并持续监控供应链风险。
